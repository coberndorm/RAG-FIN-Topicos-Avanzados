# FIN-Advisor: Inputs & Outputs Definition

---

## INPUTS

Inputs are the data sources that feed the RAG pipeline at query time. They are divided into two categories: user queries (the trigger) and transactional data (the context).

### Input 1: User Query (Natural Language)

| Attribute | Detail |
|---|---|
| Source | Chat interface in the EverGreen FIN module |
| Format | Free-text string in Spanish |
| Max Length | 500 characters (soft limit, enforced by frontend) |
| Language | Spanish (Colombian) |
| Required | Yes — this is the trigger for every interaction |

**Conditions:**
- The system must handle informal language, typos, and colloquial expressions
- Queries outside the defined scope must be gracefully rejected with a helpful message
- Empty or nonsensical queries should prompt the user to rephrase

---

### Input 2: Accounting Movements (Movimientos Contables)

| Attribute | Detail |
|---|---|
| Source | EverGreen FIN relational database (mocked in SQLite/PostgreSQL) |
| Format | Structured records (rows in a table) |
| Access | Read-only via `query_evergreen_finances` tool |
| Refresh | Real-time query at each request (no caching for MVP) |

**Data schema:**

| Field | Type | Description |
|---|---|---|
| id | int | Unique movement identifier |
| date | date | Transaction date |
| type | enum | `INGRESO` (income) or `EGRESO` (expense) |
| category | string | Category (e.g., "Insumos", "Venta cosecha", "Servicios") |
| amount | decimal | Amount in COP |
| description | string | Free-text description of the movement |
| account_id | int | FK to the chart of accounts |

**Conditions:**
- Data must cover at least the last 6 months for meaningful analysis
- All amounts are in COP (Colombian Pesos)
- The agent must never modify this data — strictly read-only access

### Input 3: Sales Invoices (Facturación de Venta)

| Attribute | Detail |
|---|---|
| Source | EverGreen FIN relational database |
| Format | Structured records |
| Access | Read-only via `query_evergreen_finances` tool |

**Data schema:**

| Field | Type | Description |
|---|---|---|
| invoice_id | int | Unique invoice identifier |
| date_issued | date | Date the invoice was created |
| date_due | date | Payment due date |
| client_name | string | Buyer name |
| total_amount | decimal | Total invoiced amount in COP |
| vat_amount | decimal | VAT component of the invoice |
| status | enum | `PAID`, `PENDING`, `OVERDUE` |

**Conditions:**
- Pending and overdue invoices are critical for cash flow projections
- The system must be able to calculate total receivables from pending invoices

## OUTPUTS

Outputs are the responses generated by the system. Each output type corresponds to one or more user stories.

### Output 1: Financial Diagnostic Report

| Attribute | Detail |
|---|---|
| Triggered by | US-02 (Cash Flow Diagnosis), US-06 (Expense Optimization) |
| Format | Structured natural language in the chat interface |
| Language | Spanish |

**Structure:**
```
1. Summary: One-sentence overview of the financial situation
2. Key Findings: Top 3 factors explaining the current state
   - Each finding includes: category, amount, and explanation
3. Comparison: Current period vs. previous period (if data available)
4. Recommendations: 2-3 actionable suggestions
5. Disclaimer: "This analysis is based on registered data. Consult a professional for formal decisions."
```

**Conditions:**
- Must include actual monetary figures from the user's data
- Must explain causes, not just list numbers
- Must always include the disclaimer
- Maximum response length: 400 words

---

### Output 2: Personalized Tax Calendar

| Attribute | Detail |
|---|---|
| Triggered by | US-04 (Personalized Tax Calendar) |
| Format | Chronological list in the chat interface |
| Language | Spanish |

**Structure:**
```
Upcoming Tax Obligations (next 3 months):

1. [DATE] — [OBLIGATION NAME]
   Estimated amount: $X,XXX,XXX COP
   ⚠️ Warning: Projected balance on this date may be insufficient

2. [DATE] — [OBLIGATION NAME]
   Estimated amount: $X,XXX,XXX COP
   ✅ Projected balance is sufficient
```

**Conditions:**
- Dates must be in chronological order
- Each entry must include the obligation name and estimated amount
- Liquidity warnings must be based on projected balance (current balance + expected receivables - expected payables)
- Must specify that only national (DIAN) obligations are covered

---

### Output 3: Investment Viability Recommendation

| Attribute | Detail |
|---|---|
| Triggered by | US-03 (Fixed Asset Purchase Viability) |
| Format | Structured recommendation in the chat interface |
| Language | Spanish |

**Structure:**
```
1. Asset: [Name and estimated cost]
2. Tax Benefits: [Applicable deductions/discounts with article references]
3. Effective Cost: [Cost after tax benefits]
4. Financial Snapshot:
   - Current balance: $X
   - Pending receivables (next 30 days): $X
   - Pending payables (next 30 days): $X
   - Available for investment: $X
5. Verdict: [VIABLE / NOT VIABLE NOW / VIABLE IN X WEEKS]
6. Reasoning: [Explanation of the calculation]
7. Disclaimer
```

**Conditions:**
- Must show the full calculation transparently
- Must reference specific tax articles if benefits apply
- If not viable now, must suggest the earliest viable date
- Must always include assumptions used in the projection

---

### Output 4: Tax Benefit Explanation

| Attribute | Detail |
|---|---|
| Triggered by | US-01 (Tax Benefit Inquiry) |
| Format | Explanatory text in the chat interface |
| Language | Spanish |

**Structure:**
```
1. Applicable Benefit: [Name and summary]
2. Legal Basis: [Article number and brief quote]
3. Conditions: [Requirements the producer must meet]
4. Estimated Impact: [How much they could save, if calculable]
5. Recommendation: [Next steps to take advantage of the benefit]
6. Disclaimer
```

**Conditions:**
- Must always cite the specific article number
- Must explain conditions in plain language
- Must include the disclaimer about professional validation

---

### Output 5: Concept Explanation

| Attribute | Detail |
|---|---|
| Triggered by | US-05 (Accounting Concept Explanation) |
| Format | Short explanatory text in the chat interface |
| Language | Spanish |

**Conditions:**
- Maximum 200 words
- No unexplained jargon
- Includes a concrete example (from user data if available, otherwise generic agricultural example)
- Does not require a disclaimer (educational, not advisory)

---

## Output Quality Rules (Apply to ALL outputs)

1. **Grounding:** Every factual claim must be traceable to either the knowledge base or the user's financial data. No hallucinated figures.
2. **Transparency:** When the system is uncertain or lacks data, it must say so explicitly.
3. **Consistency:** Monetary figures must use COP with thousand separators (e.g., $12.500.000 COP).
4. **Tone:** Technical but accessible — written for a farm administrator, not an accountant.
5. **Language:** All responses in Spanish, matching the user's input language.
6. **Disclaimers:** Any output that could be interpreted as financial or legal advice must include a disclaimer.
